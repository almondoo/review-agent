# Changelog

All notable changes to this project are documented here.

## Strategy

Changelogs are maintained as hand-curated per-wave summaries at this root
level, reconciled from the per-package Changeset files in `.changeset/`.
Individual package changelogs (`packages/*/CHANGELOG.md`) are generated by
`pnpm changeset version` and are the authoritative per-package source.
This root file summarises the operator-visible highlights per release wave.

The format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).
This project follows [Semantic Versioning](https://semver.org/) from v1.0.0
onwards.

> **Note on version dates and tags**: the project does not yet cut tagged
> formal releases (`git tag` is empty, root `package.json` is `0.0.0`, all
> packages are `private: true`). Each version's date below is the landing date
> of its development wave (the `.changeset/*-wave.md` file); the comparison-URL
> tags at the bottom are created at formal-release time (see
> [docs/release-process.md](./docs/release-process.md)) and resolve once tagged.

## [1.3.0] — 2026-06-04

Wave C — GitHub App dashboard onboarding.

### Added

- GitHub App onboarding flow: `GET /github/install-redirect` OAuth entry point,
  `GET /github/setup` callback (CSRF state-cookie verification, installation
  lookup, dashboard redirect).
- `github_installations` table with RLS tenant isolation; `repos.installation_id`
  FK (nullable, backward-compatible).
- Webhook handler extended for `installation.{created,deleted,suspend,unsuspend}`
  events.
- `GET /api/github/installations/:id/repos` and `POST /api/github/repos/bulk`
  API endpoints.
- Web pages: Integrations connect button, GitHub setup pending page, and the
  `GithubReposPage` filterable bulk-register UI.
- Multi-tenant fail-closed interlock: all `installationId`-scoped endpoints now
  reject requests whose bearer token tenant does not match the path parameter.

### Migration

Three new Drizzle migrations: `0006_github_installations.sql`,
`0007_repos_installation_id.sql`, `0008_github_installations_count_policy.sql`.
Forward-compatible — v1.2 code continues to run after the migration is applied.
New env vars: `GITHUB_APP_SLUG`, `REVIEW_AGENT_DASHBOARD_ORIGIN`.

## [1.2.0] — 2026-05-19

Wave — continuous review evaluation & improvement loop + audit cleanup.

### Added

- Closed-loop evaluation pipeline: promptfoo → severity-scoring shim,
  `baseline-measure` CLI, and unconditional severity-consistency CI gate.
- `review_eval_event` table (per-review metrics, RLS-isolated) +
  `EvalMiddleware` + `cost_ledger.latency_ms`. Fail-open so a transient DB
  outage never aborts a successfully-posted review.
- `createFeedbackWriter` (PII redact, 10/job rate-limit) + GitHub reaction /
  dismissed-signal classification + `createReviewHistoryWriter` + TTL prune
  (180-day, advisory-lock-elected).
- `<learned_facts>` injection in `composeSystemPrompt` + feedback-aware dedup.

### Changed / Fixed

- URL allowlist refined (`privacy.allowed_url_prefixes` + own-repo auto-allow).
- `privacy.deny_paths` applied to `read_file` / `glob` / `grep` dispatch.
- `privacy.redact_patterns` injected as gitleaks custom rules.
- `reviews.{path_filters, max_files, max_diff_lines}` enforced before LLM call.
- `repo.{submodules, lfs}` wired into `ProvisionWorkspaceInput.cloneHints`.
- Dead code removed: `shiftLineThroughHunks` + `ApplyLineShiftInput`.

### Migration

New migration `0003_review_eval_event.sql`. Forward-compatible — v1.1 code
keeps running after the migration is applied.

## [1.1.0] — 2026-05-16

Wave — `REQUEST_CHANGES` severity threshold, CodeCommit adapter, incremental
diff, and gitleaks secret-scan integration.

### Added

- `reviews.request_changes_on` config field; GitHub adapter now posts
  `REQUEST_CHANGES` when findings meet or exceed the configured threshold
  (default: `critical`).
- AWS CodeCommit adapter (`platform-codecommit`) — Postgres-canonical state.
- Incremental diff (`computeDiffStrategy`) — sends only the delta diff to the
  LLM on 2nd+ pushes, reducing per-PR cost.
- gitleaks integration: pre-LLM diff scan + post-LLM output scan with
  `applyRedactions`; aborts on high-confidence secret detection.
- Hidden state-comment dedup with HMAC-chained `review_state` fingerprints.
- Injection-guard middleware (LLM-based, fail-closed).
- Cost-guard middleware with per-PR `cost-cap-usd` hard cap.

## [1.0.0] — 2026-04-30

Initial stable release.

### Added

- GitHub Action entry point posting inline comments + summary comment per review.
- Vercel AI SDK agent loop with `read_file` / `glob` / `grep` tools, bounded by
  `MAX_TOOL_CALLS`.
- `.review-agent.yml` config schema (language, profile, provider/model, ignored
  authors, path instructions, skills, confidence floor).
- Multi-provider support: Anthropic (default), OpenAI, Azure OpenAI, Google,
  Vertex, Bedrock, OpenAI-compatible.
- Hono webhook server with Lambda + SQS architecture.
- CLI (`review-agent`) for local or CI invocation.
- Zod-validated output schema with `category`, `confidence`, `ruleId`,
  `severity` fields per finding.
- Non-root sandboxed Docker container with denylist enforced at provisioner and
  tool dispatcher.
- `audit_log` with HMAC chain for tamper detection.
